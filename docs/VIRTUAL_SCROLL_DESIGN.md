# Liked Media Masonry 完全仮想スクロール設計案

## 目的

数千件規模のメディアを閲覧した場合でも、カード外枠を含むDOM数、画像デコード量、レイアウト計算量を一定範囲に抑える。現行のIntersectionObserverによる内容解放は維持しつつ、カード要素そのものを表示範囲周辺だけに限定する。

この文書は設計案であり、v0.14.1では完全仮想スクロールを実装しない。

## 現行構造

- `state.items`が取得順の全メディアを保持する。
- マソンリー列ごとにカード要素を配置する。
- 画面外カードは画像やホバー情報を解放するが、カード外枠はDOM上に残る。
- カードの推定高さは、取得済み比率または過去の実測値から算出する。
- 列幅や列数が変わった場合は全カードの所属列を再計算する。

この方式は数百件程度では扱いやすいが、数千件ではDOMノード、IntersectionObserver対象、レイアウトツリーが増え続ける。

## 採用する方式

### 1. 論理レイアウトとDOM表示を分離する

全項目についてDOMを作らず、次の論理レイアウト情報だけを保持する。

```text
itemKey
columnIndex
logicalTop
logicalHeight
logicalBottom
measuredHeight
```

列ごとの論理高さは累積値として保持し、カード追加時は最も低い列へ配置する。カード幅または列数が変わった場合だけ全論理配置を再計算する。

### 2. 列ごとのスペーサーを使用する

各列は次の3領域で構成する。

```text
上側スペーサー
表示中カード群
下側スペーサー
```

表示範囲外のカード高さはスペーサーへ合算する。これによりスクロール全長を維持しながら、実DOMカード数を制限する。

### 3. 表示ウィンドウ

初期案では、ビューポート上下それぞれ2画面分を余白として実DOM化する。

```text
windowTop    = scrollTop - viewportHeight * 2
windowBottom = scrollTop + viewportHeight * 3
```

列ごとの論理カード配列を二分探索し、`logicalBottom >= windowTop`から`logicalTop <= windowBottom`までを表示対象にする。

### 4. DOM再利用

カード要素を単純に破棄・再作成せず、種類別の小さなプールを持つ。

- 画像カード
- 動画カード
- プレースホルダーカード

再利用時はイベントリスナーをカード単位で増やさず、グリッド上のイベント委譲を使用する。`data-item-key`だけを差し替え、現在の項目を`itemMap`から解決する。

### 5. 高さ補正

画像読込後に実測高さが推定値と異なる場合は、そのカード以降の同列項目だけを補正する。

- 差分が1px未満なら無視する。
- 表示中の基準カードをアンカーとして保持する。
- 補正後もアンカーの画面内位置が変わらないよう、ボードの`scrollTop`を差分調整する。
- ライトボックス表示中に利用者が背景を手動スクロールした場合は、自動同期で位置を戻さない。

## 状態案

```javascript
virtualLayout: {
  enabled: false,
  columnCount: 0,
  columnWidth: 0,
  columns: [],
  positionsByKey: new Map(),
  mountedKeys: new Set(),
  renderWindowTop: 0,
  renderWindowBottom: 0,
  updateRaf: 0,
  anchorKey: '',
  anchorViewportOffset: 0,
}
```

`items`、`itemMap`、`itemIndexMap`は現行どおり正本とし、仮想レイアウトは表示用派生状態として扱う。

## ライトボックスとの整合

- ライトボックスの現在項目は`itemKey`で保持し、DOMカードの有無に依存させない。
- 前後移動は`itemIndexMap`からO(1)で現在位置を取得する。
- 閉じる際に元カードが仮想範囲外なら、そのカードを含む位置へ先にスクロールし、マウント完了後にフォーカスを戻す。
- 背景を手動スクロールした後は、前後移動で背景位置を自動同期しない。

## 追加読込との整合

- 新規項目は論理列へ差分追加し、既存項目を再配置しない。
- 読込中のライトボックス項目と再生セッションは維持する。
- 下端判定は実DOMの末尾ではなく、論理全高と`scrollTop`から計算する。
- X側の追加読込とボード仮想化は別Controllerとし、相互にキャンセル状態を混在させない。

## 動画との整合

- 仮想範囲外へ出た動画カードはサムネイルDOMを解放する。
- X所有videoの借用状態はカードDOMから独立して管理する。
- 強参照上限3件と、現在・前後動画優先の方針を維持する。
- 仮想カードのアンマウントを理由に、ライトボックス再生中のvideoを停止しない。

## 段階導入

1. 論理位置計算だけを追加し、現行DOM配置結果と一致することをテストする。
2. 1列構成でスペーサー方式を導入する。
3. 複数列へ拡張する。
4. フォーカス復帰、ライトボックス同期、追加読込を統合する。
5. 500件、2,000件、5,000件の模擬データで計測する。
6. 実際のX上でSPA移動、動画、追加読込を検証してから既定有効化する。

## 回帰テスト要件

- 縦長画像の比率と高さが維持される。
- 表示ウィンドウ移動後も順序が変わらない。
- 同一メディアのDOMが二重にマウントされない。
- ライトボックスの前後移動に画面外項目を含む。
- 閉じた際に元カードへフォーカスが戻る。
- 背景スクロール後の位置を勝手に戻さない。
- 追加読込中も現在のメディアと動画再生を維持する。
- 列数変更後もスクロールアンカーを維持する。
- 5,000件で実DOMカード数が表示窓に比例し、全件数に比例しない。

## 性能目標

模擬データ5,000件、デスクトップ幅1920pxを目安とする。

- 実DOMカード数: 300件未満
- スクロール中の長時間タスク: 原則50ms未満
- 仮想範囲更新: 1フレームにつき最大1回
- レイアウト再計算: 列数・カード幅変更時を除き差分更新
- 画面外画像・動画: デコード済みリソースを保持し続けない

## 既知の難所

- マソンリーは列ごとの高さが異なるため、単一リストよりアンカー補正が複雑になる。
- 画像の実測高さ確定後に列全体の後続位置が変わる。
- フォーカス対象が未マウントの場合、スクロールとDOM生成を二段階で行う必要がある。
- XのDOM交換と拡張側の仮想化を同時に扱うため、実際のX上での長時間確認が不可欠である。
